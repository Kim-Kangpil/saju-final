# 모바일 카카오 로그인 시 "목록/로그인 정보 불러오지 못함" 원인

## 원인 한 줄 요약

**프론트(예: hsaju.com)와 백엔드(예: onrender.com)가 서로 다른 도메인이라서, 모바일 브라우저가 로그인 쿠키를 백엔드로 보내지 않기 때문입니다.**

---

## 왜 그런 일이 생기나

1. **로그인 직후**
   - 사용자가 카카오 로그인을 누르면 → 카카오 → **백엔드**(onrender.com) 콜백으로 옴.
   - 백엔드가 `hsaju_session` 쿠키를 **Set-Cookie**로 넣어 주지만, 이 쿠키는 **백엔드 도메인(onrender.com)** 소속으로 저장됩니다.

2. **이후 사주 목록/로그인 정보 조회**
   - 사용자는 **프론트(hsaju.com)** 에 있고, 프론트가 **백엔드(onrender.com)** 로 `/api/me`, `/api/saju/list` 등을 호출합니다.
   - 이때 요청은 **hsaju.com → onrender.com** 이라서 **다른 사이트(크로스 사이트) 요청**입니다.
   - `hsaju_session` 쿠키는 **onrender.com** 에 대한 쿠키이므로, 브라우저 입장에서는 "다른 사이트(onrender.com)로 가는 요청에 실어 보내는 쿠키" = **제3자 쿠키(third-party cookie)** 에 가깝습니다.

3. **모바일 브라우저 동작**
   - **Safari**, **모바일 Chrome** 등은 보안/개인정보 정책으로 **제3자 쿠키를 막거나 제한**합니다.
   - 그래서 hsaju.com 페이지에서 onrender.com 으로 요청을 보낼 때, **`hsaju_session` 쿠키가 아예 안 붙어서** 백엔드로 갑니다.
   - 백엔드는 쿠키가 없으니 "로그인한 사용자"를 알 수 없고 → **401** 또는 빈 데이터를 돌려줍니다.
   - 그래서 **사주 목록도 불러오지 못했다**, **로그인 정보도 제대로 표기 못한다** 는 현상이 납니다.

4. **PC에서는 잘 되는 경우**
   - PC 브라우저는 (설정·브라우저에 따라) 제3자 쿠키를 아직 허용하는 경우가 많아서, 같은 구조여도 쿠키가 전달될 수 있습니다.
   - 모바일만 막는 정책이 강해서 모바일에서 먼저 문제가 보입니다.

---

## 도메인/CORS 설정이 맞아도 모바일에서 안 되는 이유

- **FRONTEND_URL**, **CORS_ORIGINS** 를 맞춰도:
  - CORS는 "백엔드가 hsaju.com 에서 오는 요청을 허용할지"만 결정합니다.
  - **쿠키를 보낼지 말지는 브라우저가 정책으로 결정**합니다.
  - 모바일에서는 "다른 도메인으로 가는 요청에 쿠키 실어 보내기"를 막기 때문에, CORS가 허용돼 있어도 **쿠키가 안 붙어서** 로그인/목록이 안 됩니다.

즉, **원인은 "설정 실수"가 아니라 "프론트와 백엔드가 서로 다른 도메인 + 모바일의 제3자 쿠키 제한"** 입니다.

---

## 해결 방향

### 1) 같은 도메인으로 API 제공 (가장 확실한 방법)

- 프론트: `https://www.hsaju.com` (또는 `https://hsaju.com`)
- API: **같은 도메인**에서 제공  
  예: `https://api.hsaju.com` 이고, **쿠키를 `domain=.hsaju.com` 으로 설정**  
  또는 **프론트와 API를 같은 호스트**에서 리버스 프록시로 제공 (예: Vercel + API Routes / 같은 서버의 /api)
- 이렇게 하면 "hsaju.com(또는 api.hsaju.com) → 같은 도메인" 요청이 되어 쿠키가 **1자(first-party) 쿠키**로 취급되고, 모바일에서도 전송됩니다.
- **실제 조치**:  
  - API를 **api.hsaju.com** 등으로 붙이고,  
  - 백엔드에서 Set-Cookie 시 **domain을 `.hsaju.com`** 처럼 상위 도메인으로 두거나,  
  - 아예 프론트와 API를 같은 호스트에서 서빙하도록 구성.

### 2) 쿠키 대신 토큰(Authorization) 사용

- 로그인 성공 후 백엔드가 **세션 쿠키 대신(또는 보조로)** **짧은 유효기간의 토큰**을 URL 등으로 전달.
- 프론트는 이 토큰을 **메모리 또는 sessionStorage**에 두고, API 요청 시 **`Authorization: Bearer <토큰>`** 으로 보냄.
- 백엔드는 **쿠키가 없으면 Authorization 헤더에서 user 식별**하도록 수정.
- 이렇게 하면 "다른 도메인"이어도 **쿠키에 의존하지 않으므로** 모바일에서도 로그인/목록이 동작할 수 있습니다. (구현·보안 설계는 필요합니다.)

## 정리

| 현상 | 원인 |
|------|------|
| 모바일 카카오 로그인 후 사주 목록 불러오지 못함 | 프론트(hsaju.com)와 백엔드(onrender.com)가 달라서, 모바일에서 `hsaju_session` 쿠키가 백엔드로 전달되지 않음 |
| 로그인 정보도 제대로 표기 못함 | 위와 동일. `/api/me` 호출 시 쿠키가 없어서 서버가 사용자를 알 수 없음 |

---

## 적용된 해결: Authorization Bearer 토큰

- 로그인 성공 시 백엔드가 리다이렉트 URL **fragment**(`#t=토큰`)에 **세션 토큰**을 담아 보냅니다.
- 프론트 `login/success` 페이지에서 이 토큰을 **sessionStorage**에 저장하고, 이후 `/api/me`, `/api/saju/list`, `/api/saju/save` 등 모든 API 요청에 **`Authorization: Bearer <토큰>`** 헤더를 붙입니다.
- 백엔드는 **쿠키가 없으면** 요청 헤더의 **Bearer 토큰**을 검증해 user_id를 얻고, 동일하게 동작합니다.
- 따라서 **모바일에서 쿠키가 안 붙어도** 토큰으로 로그인·목록·저장이 가능합니다.
- 배포 시 백엔드 `.env`에 **`SESSION_TOKEN_SECRET`** 을 설정하는 것을 권장합니다(없으면 기본값 사용).
